## Summary

The earlier extraction round pulled out `ChartCard`, `ChartTooltip`, `InsightBlock`,
`RecommendationsList`, `StatCard`, `SectionTitle`, and `CohortHeatmap` — and moved page layout
(`projectHero`, `projectGrid2`, …) to global classes in `index.css`. What's **left** is three

the **hero**, the **SQL snippet card**, and the **info note** — plus the **Recharts dark-axis
boilerplate** repeated in ~12 charts. These four findings cover ~150 lines of duplicated JSX and
~120 lines of triplicated CSS. The SQL card and the chart-theme constants are the cleanest, highest-value wins.

## Finding 1 — `SqlCard` component

**Pattern:** An identical "SQL snippet" card (header with title + repo link, then a `<pre>` code block) is copy-pasted into all three pages.
**Affects:** `src/pages/Project1.jsx` (181–189), `Project2.jsx` (142–150), `Project3.jsx` (192–200) — plus the per-page `SQL_SNIPPET` const and 5 CSS classes triplicated in every module.
**Duplication:** ~9 lines JSX × 3 + `.sqlCard`/`.sqlCardHeader`/`.sqlCardTitle`/`.sqlCardLink`/`.sqlCode` (~25 lines CSS) × 3 = ~100 lines, ~70 eliminated.
**Proposed module:** `src/components/SqlCard.jsx` (+ `SqlCard.module.css`)

**What moves in:**
- The `<div className={styles.sqlCard}>` → header (`title`, link with `↗`) → `<pre>{snippet}</pre>` markup
- The 5 `sqlCard*` / `sqlCode` CSS classes (verbatim-identical across all three modules)

**What stays per-file:**
- The `SQL_SNIPPET` string (it's data — each project's actual query) and the locale keys (`pN.sql_card_title`, `pN.sql_link`) and `sqlUrl`.

**Extraction plan:**
1. Create `SqlCard.jsx` accepting `{ title, code, href, linkLabel }`.
2. Move the 5 CSS classes into `SqlCard.module.css`; delete them from all three project modules.
3. Replace each block with `<SqlCard title={t('pN.sql_card_title')} code={SQL_SNIPPET} href={sqlUrl} linkLabel={t('pN.sql_link')} />`.

**Why it matters:** A change to the code-block styling (font, scroll behavior, header layout) currently means editing three JSX blocks and three CSS files. One wrong edit and the three projects' SQL cards drift apart.
**Testability gain:** The `<pre>` rendering and the link `target="_blank" rel="noopener"` safety attributes become testable once, in isolation.

## Finding 2 — `chartTheme.js` shared Recharts axis/grid constants

**Pattern:** Every chart repeats the same dark-theme axis and grid configuration with the same three hardcoded hex colors.
**Affects:** ~12 charts across `Project1.jsx`, `Project2.jsx`, `Project3.jsx`.
**Duplication:** Each `<XAxis>`/`<YAxis>` carries `tick={{ fill: '#636e7b', fontSize: 10 }} axisLine={false} tickLine={false}`; each `<CartesianGrid>` carries `strokeDasharray="3 3" stroke="#1e2530"`. The hex values `#1e2530`, `#636e7b`, `#cdd9e5` are repeated dozens of times.
**Proposed module:** `src/utils/chartTheme.js`

**What moves in:**
```js
export const COLORS = { grid: '#1e2530', tickMuted: '#636e7b', tickStrong: '#cdd9e5' }
export const gridProps   = { strokeDasharray: '3 3', stroke: COLORS.grid }
export const axisMuted   = { tick: { fill: COLORS.tickMuted, fontSize: 10 }, axisLine: false, tickLine: false }
export const axisStrong  = { tick: { fill: COLORS.tickStrong, fontSize: 10 }, axisLine: false, tickLine: false } // category labels
```

**Usage:** `<XAxis {...axisMuted} tickFormatter={fmtUSD} />`, `<YAxis {...axisStrong} dataKey="title" width={130} />`, `<CartesianGrid {...gridProps} horizontal={false} />`. Per-chart overrides (`width`, `domain`, `tickFormatter`, `fontSize: 11`) still pass as extra props after the spread.

**What stays per-chart:** Everything chart-specific — `dataKey`, `tickFormatter`, `domain`, `width`, the data, the `<Bar>`/`<Line>` colors.

**Extraction plan:**
1. Create `src/utils/chartTheme.js` with the constants above.
2. In each chart, replace the inline `tick`/`axisLine`/`tickLine` and grid props with `{...axisMuted}` / `{...axisStrong}` / `{...gridProps}`.
3. Leave Recharts series colors (`#00cc96`, `#a371f7`, …) per-chart for now — they're accent choices, not theme tokens (see *Not recommended*).

**Why it matters:** These three greys are the chart theme. They can't live in CSS vars because Recharts needs literal JS values, so a JS constants module is their correct home. Right now a theme tweak (lighter grid line) is a find-and-replace across three files and dozens of occurrences — easy to miss one and leave a chart visually off.
**Testability gain:** Low directly (they're constants), but it makes the chart code dramatically more readable and removes the "did I update every axis?" risk.

## Finding 3 — `ProjectHero` component (hero + prev/next nav)

**Pattern:** The page top — prev/next nav buttons, then a hero with two tags, a two-line title, a subtitle, a tech-pill row, and a two-link row — is structurally identical across all three pages.
**Affects:** `Project1.jsx` (50–84), `Project2.jsx` (38–72), `Project3.jsx` (76–110). Plus `.tag`, `.heroAccent`, `.linkRow`, `.sqlLink` (+ `.pythonLink`/`.kaggleLink`) triplicated in every module.
**Duplication:** ~30 lines JSX × 3 + ~30 lines CSS × 3 = ~180 lines, ~110 eliminated.
**Proposed module:** `src/components/ProjectHero.jsx` (+ `ProjectHero.module.css`)

**What moves in:**
- The `prev-btn`/`next-btn` buttons, the `projectHero` section, the meta/tags, title, sub, tech-pill `.map`, and the `linkRow`
- The `.tag`, `.heroAccent`, `.linkRow`, and link-anchor CSS

**What stays per-file:** Pure data passed as props:
```jsx
<ProjectHero
  prev={{ id: 'home', label: t('nav.home') }} next={{ id: 'p2', label: t('nav.p2') }}
  setActive={setActive}
  tags={[{ label: t('p1.tag') }, { label: t('p1.tag2'), color: 'var(--green)' }]}
  title1={t('p1.title1')} title2={t('p1.title2')}
  sub={t('p1.sub')}
  tech={['SQL', 'Python', 'DuckDB', …]}
  links={[{ label: t('p1.python_link'), href: pythonUrl }, { label: t('p1.sql_link'), href: sqlUrl }]}
/>
```

**Honest caveat (ponytail):** This is the heaviest of the four — ~6 props, two of them arrays. It was *previously* in "Not recommended" for that reason. It earns its place now because (a) the three copies are byte-identical in structure, and (b) it also dedups four triplicated CSS classes. It's a flat presentational leaf, not prop-drilling. If you'd rather keep heroes editable in-place, take Findings 1, 2, 4 and skip this one — that's a defensible call.

**Why it matters:** Adding a third hero link or changing the tag styling is a three-file edit today. The tag-2 accent color is already hand-copied as an inline style in each page (green/green/purple) — exactly the kind of thing that drifts.
**Testability gain:** The nav wiring (`onClick={() => setActive(next.id)}`) and the links' `rel="noopener"` become testable once.

## Finding 4 — `Note` component (info callout)

**Pattern:** A small "ℹ + HTML paragraph" note box. Same structure in three places, but named inconsistently.
**Affects:** `Project1.jsx` (234–237, classes `mlNote`/`mlNoteIcon`), `Project2.jsx` (138–141 and 184–187, classes `note`/`noteIcon`).
**Duplication:** ~4 lines × 3 instances + two near-identical CSS class pairs (`note`/`mlNote`). ~35 lines, ~20 eliminated.
**Proposed module:** `src/components/Note.jsx` (+ `Note.module.css`)

**What moves in:**
```jsx
<div className={styles.note}>
  <span className={styles.noteIcon}>ℹ</span>
  <p dangerouslySetInnerHTML={{ __html: text }} />
</div>
```
**Props:** `text` (HTML string).
**What stays per-file:** The locale key (`t('p1.ml_note')`, `t('p2.note_retention')`, …).

**Extraction plan:**
1. Create `Note.jsx` accepting `{ text }`.
2. Reconcile `mlNote`/`mlNoteIcon` (P1) and `note`/`noteIcon` (P2) into one `Note.module.css` — they're the same visual, the `ml` prefix is incidental.
3. Replace all three blocks with `<Note text={t('…')} />`.

**Why it matters:** This is the one place besides `InsightBlock`/`RecommendationsList` that renders `dangerouslySetInnerHTML` — centralizing it means the HTML-injection surface lives in three named components, not scattered inline blocks.
**Testability gain:** The HTML rendering is verified once; the `mlNote`/`note` naming drift disappears.

## Finding 5 — Move `.kpiRow` to a global class (CSS only)

**Pattern:** `.kpiRow` is a `StatCard` grid, defined verbatim in all three project modules and wrapped as `<section className={styles.kpiRow}>` on every page.
**Affects:** `Project1/2/3.module.css` (`.kpiRow` + its responsive rules, ~3 occurrences each).
**Duplication:** ~10 lines CSS × 3 = ~30 lines, ~20 eliminated. **No new component** — it's a layout grid like `.projectGrid2`, which already lives in `index.css`.

**Extraction plan:**
1. Add `.projectKpiRow` to `src/index.css` next to `.projectGrid2` (copy the grid + its `@media` rule).
2. Replace `className={styles.kpiRow}` → `className="projectKpiRow"` in all three pages.
3. Delete `.kpiRow` from the three modules.

**Why it matters:** It's the same grid-of-cards layout; it belongs with the other global `project*` layout utilities, not copied three times.
**Testability gain:** None (CSS layout) — pure DRY / single point of change.

## Not recommended

- **`KpiRow` as a JSX component.** Each page passes a different *count* of `StatCard`s (6 / 4 / 6) with project-specific labels, values, and accent colors. A wrapper component would just forward `children` around a grid — the grid is the only shared part, and Finding 5 handles that with a CSS class. A component adds indirection for zero logic.
- **Merging `ScatterTip` into `ChartTooltip`.** `ScatterTip` (P3) does a `SEGMENT_COLORS` lookup and renders three fixed, labeled rows with units (`days` / `orders` / `$`). `ChartTooltip` maps a generic payload. Unifying them means adding a "scatter mode" branch and a colors prop to `ChartTooltip` for a single call site — more complexity than the ~12 lines it saves. Leave it inline in P3.
- **Extracting Recharts *series* colors** (`#00cc96`, `#a371f7`, `#ff6b35`, …). Unlike the three axis greys, these are deliberate per-chart accent choices (P1 cyan, P2 orange, P3 green). Tokenizing them into `chartTheme.js` would imply they're interchangeable; they aren't. Keep them at the call site.
- **Unifying the data files** (`project1/2/3.js`). Different domains, different shapes (salary arrays vs A/B counters vs RFM segments). Nothing to share.
- **`prev-btn`/`next-btn` as a standalone component.** Four lines each; they fold naturally into `ProjectHero` (Finding 3). Not worth a separate module.
