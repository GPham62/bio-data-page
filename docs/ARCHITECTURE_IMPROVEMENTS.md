# Architecture Improvements

> Generated by /improve-codebase-architecture — confirm before applying.

## Summary

The three project pages (`Project1.jsx`, `Project2.jsx`, `Project3.jsx`) share a significant amount of copy-pasted code across four dimensions: a chart tooltip component (identical in all three), number formatting utilities (three slightly different copies), an insight callout markup pattern (repeated 5+ times across all three pages), and ~50 lines of CSS repeated verbatim in every module file. Extracting these reduces the bug surface for future edits — right now, a one-line change to the tooltip style requires editing three files.

---

## Finding 1 — Shared `ChartTooltip` component

**Pattern:** The `Tip` custom tooltip component is copy-pasted verbatim into all three project pages.

**Affects:** `src/pages/Project1.jsx` (lines 16–31), `src/pages/Project2.jsx` (lines 16–31), `src/pages/Project3.jsx` (lines ~18–33)

**Duplication:** ~26 lines × 3 files = 78 lines, ~52 eliminated

**Proposed module:** `src/components/ChartTooltip.jsx`

**What moves in:**
- The `Tip` component body — `active`/`payload`/`label`/`prefix`/`suffix` props, the dark tooltip shell, the label row, the payload map
- A `color` prop (default `'#00e5ff'`) to replace the hardcoded fallback — P3 currently defaults to `#00cc96` and `ScatterTip` is a minor variant

**What stays per-file:**
- Nothing — each file just does `import ChartTooltip from '../components/ChartTooltip.jsx'` and passes its accent color

**Extraction plan:**
1. Create `src/components/ChartTooltip.jsx` accepting `{ active, payload, label, prefix, suffix, color }` where `color` defaults to `'#00e5ff'`
2. In Project1/2, replace local `Tip` with `ChartTooltip` — no color prop needed (default matches)
3. In Project3, replace local `Tip` with `ChartTooltip color="#00cc96"` and replace `ScatterTip` with `ChartTooltip color="#00cc96"` (same shape, only `name` rendering differs — reconcile into the shared component)

**Why it matters:** Every tooltip style change (padding, font size, border radius) currently requires three separate edits. One wrong edit in one file creates visual inconsistency across projects.

**Testability gain:** The tooltip's conditional rendering (`!active || !payload?.length → null`) can now be unit-tested once as a standalone component.

## Finding 2 — Shared `formatters.js` utility

**Pattern:** Number formatting functions are defined locally in all three project pages with slight variations.

**Affects:** `src/pages/Project1.jsx` (lines ~10–14), `src/pages/Project2.jsx` (line ~10), `src/pages/Project3.jsx` (lines ~10–14)

**Duplication:** ~12 lines × 3 files = 36 lines, ~24 eliminated

**Proposed module:** `src/utils/formatters.js`

**Current state:**
- `fmt` in P1/P2: handles only `K` — `v >= 1000 ? ${(v/1000).toFixed(0)}k : v`
- `fmt` in P3: handles `M` then `K` — the better version
- `fmtUSD` in P1: `$${(v/1000).toFixed(0)}k` — hardcodes K, breaks for millions
- `fmtUSD` in P3: `$${fmt(v)}` — delegates to `fmt`, inherits M/K handling

**What moves in:**
```js
export const fmt    = (v) => v >= 1_000_000 ? `${(v/1_000_000).toFixed(1)}M` : v >= 1_000 ? `${(v/1_000).toFixed(0)}k` : v
export const fmtUSD = (v) => `$${fmt(v)}`
```

**What stays per-file:** Nothing — each file imports the functions it uses.

**Extraction plan:**
1. Create `src/utils/formatters.js` with the consolidated `fmt` (P3 version) and `fmtUSD`
2. In each project page, replace the local definitions with `import { fmt, fmtUSD } from '../utils/formatters.js'`
3. The P1/P2 `fmt` change is a no-op for existing values (all their numbers are < 1M)

**Why it matters:** P1's `fmtUSD` silently breaks if the salary data ever includes values above $999K — it would render `$NaN k` instead of `$1.2M`. The P3 version is already correct.

**Testability gain:** Both formatters become independently testable pure functions.

## Finding 3 — `InsightBlock` component

**Pattern:** Every content section ends with an identical 5-line JSX block for the insight callout.

**Affects:** `src/pages/Project1.jsx` (5 occurrences), `src/pages/Project2.jsx` (4 occurrences), `src/pages/Project3.jsx` (4 occurrences) — 13 instances total

**Duplication:** ~5 lines × 13 instances = 65 lines, ~52 eliminated

**Proposed module:** `src/components/InsightBlock.jsx`

**What moves in:**
```jsx
<div className={styles.insight}>
  <span className={styles.insightLabel}>{label}</span>
  <p className={styles.insightText} dangerouslySetInnerHTML={{ __html: text }} />
</div>
```

**Props:** `label` (e.g. `t('p1.insight_label')`) and `text` (e.g. `t('p1.s1_insight')`)

**What stays per-file:** The locale key calls — each page still passes its own translated strings.

**Extraction plan:**
1. Create `src/components/InsightBlock.jsx` accepting `{ label, text }` — renders with its own `styles.insight` etc. from a new `InsightBlock.module.css`
2. Cut shared CSS (`.insight`, `.insightLabel`, `.insightText`, `.insightText strong`) out of all three project CSS modules and into `InsightBlock.module.css`
3. Replace every `<div className={styles.insight}>` block with `<InsightBlock label={...} text={...} />`

**Why it matters:** The insight block currently hardwires the accent color via per-project CSS. Right now changing the block's layout (e.g. adding a bullet icon) requires editing all three modules. Extraction centralises that.

**Testability gain:** The `dangerouslySetInnerHTML` rendering can now be tested once (including verifying that XSS-safe HTML is required, since this pattern is the one place in the codebase that renders arbitrary HTML).

## Finding 4 — `RecommendationsList` component

**Pattern:** The numbered recommendations box is structurally identical in Project1 and Project3 (different accent colors, same markup).

**Affects:** `src/pages/Project1.jsx` (recs section), `src/pages/Project3.jsx` (recs section)

**Duplication:** ~15 lines × 2 files = 30 lines, ~15 eliminated

**Proposed module:** `src/components/RecommendationsList.jsx`

**What moves in:**
```jsx
<div className={styles.recsBox}>
  <h3 className={styles.recsTitle}>{title}</h3>
  <ul className={styles.recsList}>
    {items.map((item, i) => (
      <li key={i} className={styles.recsItem}>
        <span className={styles.recsNum}>0{i + 1}</span>
        <span dangerouslySetInnerHTML={{ __html: item }} />
      </li>
    ))}
  </ul>
</div>
```

**Props:** `title` (string), `items` (string[], each may contain `<strong>`)

**What stays per-file:** Locale lookups — `t('p1.recs_title')` and `t('p1.recs_items', { returnObjects: true })` stay in the page.

**Extraction plan:**
1. Create `src/components/RecommendationsList.jsx` with props `{ title, items }`
2. Move `recsBox` / `recsTitle` / `recsList` / `recsItem` / `recsNum` CSS from both P1 and P3 modules into `RecommendationsList.module.css` — use a `--recs-accent` CSS custom property so each project can pass its accent color if needed
3. Replace the JSX blocks in both pages with `<RecommendationsList title={...} items={...} />`

**Why it matters:** P2 uses a `.conclusion` box for the same narrative role. If someone later adds recommendations to P2, they'd have to discover the existing component exists in the other two pages and hope to use it — or accidentally invent a fourth pattern.

**Testability gain:** The numbered badge rendering and HTML interpolation are independently verifiable.

## Finding 5 — Extract `CohortHeatmap` from Project3.jsx

**Pattern:** `CohortHeatmap` is a self-contained 45-line component defined inline at the top of `Project3.jsx`.

**Affects:** `src/pages/Project3.jsx` (lines 62–107 approximately)

**Duplication:** 0 — only one copy exists, but it belongs in `src/components/` not inline in a page file.

**Proposed module:** `src/components/CohortHeatmap.jsx`

**What moves in:** The entire `CohortHeatmap` function plus its CSS classes (`.heatmapWrap`, `.heatmap`, `.heatmapRow`, `.heatmapLabel`, `.heatmapCell`, `.heatmapHeader`, `.heatmapHeaderCell`, `.heatmapTitle`, `.heatmapSub`) — currently in `Project3.module.css`

**What stays in Project3.jsx:** The `cohortData` import and the `<CohortHeatmap data={cohortData} label={t('p3.cohort_label')} />` call.

**Extraction plan:**
1. Create `src/components/CohortHeatmap.jsx` — move the component body verbatim, create `CohortHeatmap.module.css` with the heatmap-specific classes
2. Remove the inline component and CSS from `Project3.jsx` and `Project3.module.css`
3. Add the import to `Project3.jsx`

**Why it matters:** The component is 45 lines with its own colour interpolation logic — it's not page logic, it's a reusable chart component. Mixing it with page layout makes Project3.jsx harder to read and harder to test.

**Testability gain:** The heatmap's colour interpolation (`rgba(0, 204, 150, ${opacity})`) becomes testable in isolation.

## Finding 6 — Shared project page CSS

**Pattern:** ~50 lines of CSS rules are verbatim-identical across all three project CSS modules.

**Affects:** `src/pages/Project1.module.css`, `src/pages/Project2.module.css`, `src/pages/Project3.module.css`

**Duplication:** ~50 lines × 3 = 150 lines, ~100 eliminated

**Identical blocks:**
```css
.page { max-width: 1200px; margin: 0 auto; padding: 0 32px 80px; }
.hero { padding: 56px 0 40px; border-bottom: 1px solid var(--border); margin-bottom: 48px; }
.heroMeta { display: flex; gap: 8px; margin-bottom: 20px; }
.heroTitle { font-family: var(--head); font-size: clamp(36px, 6vw, 64px); font-weight: 800; line-height: 1.05; letter-spacing: -0.03em; color: var(--text); margin-bottom: 16px; }
.heroSub { max-width: 640px; color: var(--muted); font-size: 12px; line-height: 1.8; margin-bottom: 24px; }
.heroStack { display: flex; flex-wrap: wrap; gap: 6px; }
.pill { font-size: 10px; letter-spacing: 0.08em; color: var(--muted); background: var(--bg3); border: 1px solid var(--border); border-radius: 100px; padding: 3px 10px; }
.section { margin-bottom: 60px; }
.grid2 { display: grid; grid-template-columns: 1fr 1fr; gap: 16px; }
.grid1 { display: grid; grid-template-columns: 1fr; gap: 16px; }
.insightText { font-size: 16px; line-height: 1.8; color: var(--text); margin: 0; }
```

**Proposed approach:** Move these into `src/pages/project-shared.module.css` and use CSS Modules `composes`:
```css
/* In Project1.module.css */
.page { composes: page from './project-shared.module.css'; }
```

Or simpler: move the layout-only classes (`.page`, `.section`, `.grid1`, `.grid2`) to global utility classes in `src/index.css` and drop the `.module.css` declarations entirely.

**Caution:** CSS Modules `composes` across files works in Vite but produces slightly unusual class name output. Recommend the global-utility approach for layout classes only, and leave accent-colored classes (`.tag`, `.heroAccent`, `.insight`, `.recsNum`) per-module since they need different values per project.

**Extraction plan:**
1. Add `.projectPage`, `.projectSection`, `.grid2`, `.grid1` to `src/index.css` as global classes
2. In each project JSX, replace `className={styles.page}` → `className="projectPage"`, etc.
3. Delete the now-redundant declarations from all three `.module.css` files

**Why it matters:** Adding a new grid layout (e.g. `grid3`) currently means adding it to three files. Shared layout utilities only need one edit.

**Testability gain:** Low — CSS layout isn't unit-tested. Main benefit is DRY and single point of change.

## Not recommended

**`ProjectHero` component:** The hero section JSX is structurally similar across pages, but each hero has different link combinations (P1: one Colab link; P2: Kaggle + Colab in a `linkRow`; P3: Kaggle + Colab in a `linkRow`). A shared `ProjectHero` component would need 8+ props (`tag1`, `tag2`, `title1`, `title2`, `accent`, `sub`, `pills`, `links`) and is borderline prop-drilling. The 30 lines per page are readable as-is; extraction would add more indirection than it removes.

**`KpiRow` component:** Each page uses `StatCard` with different counts (6, 4, 6) and different locale keys. Wrapping this in a `KpiRow` component would just add a thin wrapper around an already-thin pattern. Not worth it.

**Data file consolidation:** `project1.js`, `project2.js`, `project3.js` export completely different shapes matched to their domain (salary data, A/B test counters, RFM segments). Nothing to unify.

**`linkRow` CSS:** Identical in P2 and P3 but not P1 — two instances is below the threshold for extraction unless `ProjectHero` is extracted (in which case it moves there automatically).
